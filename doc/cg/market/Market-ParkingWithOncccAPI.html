2. 接口

HTTP错误响应统一示例
{
  "code" : 0, // 0：成功，
                -1：失败，
                -3：异常，
              1001：商位处于黑名单，
              1002：车牌处于黑名单，
              1003：泊车位申请批次时间有误
  "msg" : "提示信息",
  "date" : "请求返回数据"
}
接口格式
1. 接口 Header 均带上Token ：Authorization:token
2. 接口均采用 application/json 格式
统一状态对应表
申请状态表(applyStatus)：
-----------------------------
| 状态           | 状态意义   |
-----------------------------
| REVIEW        |   审核中   |
| REVIEW_YES    |  审核通过  |
| REVIEW_NOT    | 审核不通过 |
| RECEIVED_NO   |   未审领  |
| RECEIVED_YES  |   已审领  |
----------------------------
中签状态表(checkStatus)：
-------------------------
| 状态	      | 状态意义 |
-------------------------
| INIT        |  初始状态 |
| CHECK_NO	  |  未中签  |
| CHECK_YES	  |  已中签  |
-------------------------
支付状态表(payStatus)：
------------------------
| 状态	     | 状态意义  |
------------------------
| INIT       |  初始状态 |
| PAY_NO	 |  未付款  |
| PAY_YES	 |  已付款  |
------------------------
停车场状态表(parkingStatus)：
------------------------
| 状态       | 状态意义  |
------------------------
| EFFECTIVE |   有效   |
| FAILURE   |   失效   |
------------------------

流程：
1.发起申请->审核中->审核不通过->重新申请
2.发起申请->审核中->审核通过->（待中签）->未中签（结束泊车位流程）
3.发起申请->审核中->审核通过->（待中签）->已中签->待付款->超时（结束泊车位流程）
4.发起申请->审核中->审核通过->（待中签）->已中签->待付款->已付款->待审领
5.发起申请->审核中->审核通过->（待中签）->已中签->待付款->已付款->待审领->未审领/已审领
